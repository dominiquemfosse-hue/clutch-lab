❓ Mode d'emploi

Le cockpit en 6 étapes (clique)
  1. 1 · Backend. Deux choix : Mode démo locale (rien à configurer, ville émulée — pour comprendre l'outil) ou Connecter staging (les bots écrivent de VRAIES lignes dans ton Supabase de test). Jamais la prod.
  2. 2 · Population. Choisis le nombre de bots + la seed (même seed = même film, indispensable pour rejouer un bug), puis ➕ Créer les bots.
  3. 3 · Scénario. Règle les probas de comportement : ouvrir une Fenêtre, envoyer un Clutch, accepter/refuser. C'est ton panel statistique : zone creuse = peu de bots, Lausanne samedi 20h = beaucoup.
  4. 4 · ▶️ Lancer la soirée. À chaque tick les bots vivent : Fenêtres, Clutchs, Verrous, Événements — via les mêmes portes (RPC) que les humains. La carte s'anime, le journal raconte tout.
  5. 5 · Observe. Métriques live + courbe Fenêtres/Verrous = la liquidité (à quelle densité la ville « prend »). Le COQ crie en rouge si un invariant casse (double-booking, RDV infaisable, boîte ♀ saturée…) avec tick + seed pour rejouer.
  6. 6 · Creuse. Clique un bot sur la carte = POV (sa fiche, ses clutchs). 💾 Export JSON = le run complet à partager sur /collab ou à me redonner pour analyse.

🔑 Débloquer admin sur TON staging
Les RPC bots sont gardées par qa_is_admin() (UUID de David/Mel en dur). Sur ton staging, ajoute TON UUID (affiché ci-dessus après connexion) :
-- SQL Editor de TON staging :
create or replace function public.qa_is_admin()
returns boolean language sql stable as $$
  select auth.uid() = any(array[
    'TON-UUID-ICI'
  ]::uuid[]);
$$;
